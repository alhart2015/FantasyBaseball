- [ ] **Auto-compute SGP denominators from Monte Carlo** — Instead of static denominators, simulate many drafts and seasons, sort each category's team totals, and compute the average gap between adjacent teams. That gap is the denominator by definition. Self-calibrating when league size or stat environment changes. Tradeoff: only as good as the projections and draft model.

Pull actual ER/BB/H_allowed per-IP rates from historical game logs and compute the real pairwise correlations. These should differ (walks and hits allowed have different causes).

- [ ] **Clean up "default for backwards compatibility" and other code smells** — Grep for remaining `# default for backwards compat` comments and similar shims. These were added during refactors and should be resolved — either the new behavior is correct (remove the comment and dead path) or the migration is incomplete (finish it).

- [ ] **Daily summary email** — Automated morning email with: last night's results for all roster players (hits, HRs, RBI, etc.), hot/cold streaks (rolling 7-day and 14-day performance vs projection), standings changes (gained/lost roto points overnight), recommended lineup changes for today, injury news affecting roster, and upcoming probable pitcher matchups. Could use the existing leverage modules as the analytical backbone.

- [ ] **Show player ranking on other teams' rosters** — The roster view for opponents doesn't show player VAR/ranking info. Add the same ranking context shown for the user's own roster so you can quickly assess trade targets and opponent strengths.

- [ ] **Thread SGP denominator overrides through in-season code** — `league.yaml` `sgp_denominators` overrides only flow into `build_draft_board()`. All in-season call sites (`leverage.py`, `waivers.py`, `roster_audit.py`) call `get_sgp_denominators()` with no overrides, silently ignoring custom config. Currently harmless because YAML values match defaults, but will become a real bug if denominators are ever tuned. Thread `sgp_overrides` from config through all in-season entry points.

- [ ] **Calibrate DEFAULT_TEAM_IP against league data** — `DEFAULT_TEAM_IP = 1450` in `player_value.py` is likely too high for a 9-pitcher-slot league (realistic range 1200-1380 IP). This uniformly compresses ERA/WHIP SGP by ~10-15%, slightly undervaluing pitching rate stats. Pull actual team IP totals from this season's standings to calibrate. Affects both draft and in-season valuations.

- [ ] **Add per-player projection-system spread to uncertainty model** — `scoring.project_team_sds` currently applies a single per-stat CV (calibrated from 2022-2024 projection-vs-actual error) uniformly to every player. That captures irreducible outcome variance but misses that some players have tight consensus across systems (Soto HR: all 5 agree) while others have wide spreads (rookie, injury rebound, unsettled closer role) — a real uncertainty signal we already have in the raw projection files but throw away during blending. Proposal: compute per-player cross-system variance from the 5 projection CSVs during blend, persist alongside the blended mean, and combine in `project_team_sds` as `player_var = (CV * projection)^2 + spread^2`. Historical CV stays the floor (forecaster agreement ≠ certainty); cross-system disagreement adds on top. Flows into every `score_roto` caller: trade evaluator, roster audit, transaction analyzer, in-season projected standings, **and** the planned draft live-standings view. Design decision before implementation: whether to blend using inverse-variance weighting (tight-consensus players get more weight toward their mean) or keep the current fixed weights.
