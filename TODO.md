# TODO — Infrastructure / Productionization

- [ ] **Audit the path to a production-ready, multi-user architecture** — Free-tier Render + Upstash (Redis KV) is great for a single-user, single-league tool, but the longer-term goal is a multi-tenant product serving many users across both web and a mobile app. Audit what that migration takes and write up the assessment (effort, sequencing, what to keep vs. rebuild) — the deliverable is the plan, NOT the migration itself.

  **Findings from a 2026-05-22 light scoping pass (no decisions made yet):**

  Legal / ToS gates come BEFORE any infra spend — they reshape the architecture more than hosting cost does:
  - **Yahoo API ToS** (https://legal.yahoo.com/us/en/yahoo/terms/product-atos/apiforydn/): (a) Monetization needs Yahoo's prior express written permission (sec 1(e)(4)) — a paid product on the API is not allowed by default. (b) Multi-user IS contemplated — the API supports accessing "your or your users' content" (sec 1(e)(5)(iii)), so per-user OAuth is the sanctioned model. (c) **24-hour retention rule (sec 2(a)): Yahoo user data must be removed from any data repository within 24h unless the API Documents explicitly flag it storable indefinitely** — this contradicts the "store everything durably in Postgres" assumption below. Verify against the Fantasy Sports API Documents whether roster/league data is exempt; default posture is re-fetch + short-TTL cache, persist only OUR derived artifacts. (d) Single API Key per app (sec 1(e)(2)) means all users share ONE rate-limit budget (limits at Yahoo's sole discretion). (e) Attribution required: "Fantasy data provided by Yahoo Fantasy" linking back (developer.yahoo.com/attribution).
  - **FanGraphs ToS** (https://www.fangraphs.com/about/terms-of-service): redistribution and commercial use both prohibited (sec 6 unauthorized reproduction/distribution; sec 11 no commercial exploitation; sec 17 no derivative works). Pulling CSVs for personal use is a gray area we're likely fine in today; serving those projections to OTHER users in a product is squarely barred. The systems (Steamer, ZiPS, THE BAT) are also third-party — so we need a licensed/redistributable projection source before going multi-user.
  - **Cheapest decisive next steps:** email Yahoo dev relations for written permission + clarification on the 24h rule and the shared rate limit for a multi-user app; line up a redistributable projection source; time one `run_full_refresh` (MC=1000 iters) to get the per-league compute unit.

  Cost shape: the heavy data (MLB game logs, ~360MB streaks DuckDB, 50 projection files) is SHARED across all users/leagues — adding users adds megabytes, not gigabytes. What scales is per-LEAGUE compute (standings/valuations/MC run per league since categories/slots/keepers differ) and Yahoo API calls; refresh only active leagues, nightly + on-demand (data only changes when games are played). Rough hosting ballpark (verify before deciding): pilot (<=25 users) ~$25-75/mo (Render paid web + worker, managed Postgres, Upstash PAYG); hundreds of users ~$100-300/mo; thousands ~$500-2000+/mo dominated by refresh compute. Plus Apple $99/yr and Google $25 once for the app stores.

  Technical areas to cover: (1) **Durable store** — move OUR derived state (computed standings, valuations, models) and app data (users, leagues, memberships, encrypted OAuth tokens) into managed Postgres with migrations; treat raw Yahoo data as short-TTL cache per the 24h rule, NOT durable rows. App must go stateless (kill the build-time SQLite rebuild and the `RENDER=true` Upstash-vs-local-SQLite gate). (2) **Multi-tenancy** — per-user auth (buy it: Clerk/Auth0/Cognito), league isolation, per-account Yahoo OAuth token storage + refresh instead of the single `config/oauth.json`. (3) **Backend API** — Flask Jinja becomes a stable JSON API both web and mobile consume (web could stay server-rendered to halve frontend effort). (4) **Background jobs** — the refresh pipeline currently piggybacks on the web service / cron; needs a worker tier + scheduler that refreshes active leagues. (5) **Hosting & ops** — general-purpose cloud (AWS/GCP/Fly/Render paid) for horizontal scale, secrets management, observability. (6) **Mobile** — app-store accounts and push notifications (pairs with the daily-summary item above).
